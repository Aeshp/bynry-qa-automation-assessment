# Test Plan: WorkflowPro QA Automation

## Part 1
**Objective**: Validate the UI login flow and multi-tenant access controls for different user roles.

**Scope**:
- Valid login flow for an admin user.
- Multi-tenant access controls ensuring Company 2 users load the dashboard correctly.
- Cross-browser compatibility placeholder for CI execution.

**Out of Scope**:
- Real 2FA provider integration.
- Live staging environment (relies on dummy domain `app.workflowpro.com`).

**Test Environment**: Runs against `https://app.workflowpro.com` using Playwright automation. Because this is a non-resolving dummy domain and the network layer is *not* mocked for these specific tests, `test_user_login_fixed` and `test_multi_tenant_access_fixed` cannot pass locally. They are therefore explicitly skipped in execution via `@pytest.mark.skip(reason="Mock application not deployed")`. Only `test_login_cross_browser_placeholder` actually runs and passes in Part 1.

**Entry Criteria**: Python environment configured, Playwright installed, `.env` credentials loaded.

**Exit Criteria / Pass Conditions**: As executed locally via `pytest tests/`, the full suite yields 11 passed tests and 2 explicitly skipped tests. For Part 1 specifically, pass conditions mean the placeholder cross-browser test passes while the live UI tests remain safely skipped without failing the pipeline.

**Test Cases**:
| Test Name | Description |
|-----------|-------------|
| `test_user_login_fixed` | Verifies a Company 1 Admin can log in and view the dashboard. |
| `test_multi_tenant_access_fixed` | Verifies a Company 2 Employee can log in and view at least one Company 2 project card. |
| `test_login_cross_browser_placeholder` | A parameterized test to document cross-browser intent across Chromium, Firefox, and WebKit. |

## Part 2
**Objective**: Validate the framework scaffolding, fixture wiring, API client construction, and configuration loading that the rest of the suite depends on.

**Scope**:
- Validating that API client fixtures properly construct `X-Tenant-ID` and `Authorization` headers.
- Verifying that UI fixtures (`company1_admin`, `company2_employee`) correctly inject the appropriate role and tenant values into their dictionary structures.
- Confirming that Playwright's mobile viewport fixtures apply the correct CSS dimensions.

**Out of Scope**:
- Full functional RBAC enforcement or actual End-to-End UI/API behavioral checks (these tests only assert on fixture dictionary values and do not actually interact with the UI).
- Real native mobile app execution or remote BrowserStack runs.

**Test Environment**: Dummy environment. These tests do not rely on actual network responses or UI rendering; they strictly assert on the local state and header configurations generated by the framework's fixtures.

**Entry Criteria**: `pytest.ini`, `.env`, and YAML configurations properly formatted and loaded via `config_loader.py`.

**Exit Criteria / Pass Conditions**: Tests successfully prove that the scaffolding works by asserting that the HTTP headers are constructed correctly, fixtures map roles accurately into dictionary values, and viewport dimensions are set — without triggering any real UI or API interactions.

**Test Cases**:
| Test Name | Description |
|-----------|-------------|
| `test_create_project_api` | Verifies API client injects the correct `X-Tenant-ID` header during creation. |
| `test_get_project_api` | Verifies API client includes the `Authorization` header for retrieval. |
| `test_admin_can_create_project` | Validates `company1_admin` fixture injects correct role and tenant for UI creation flow. |
| `test_employee_cannot_create_project` | Validates `company2_employee` fixture injects correct employee role for restricted flows. |
| `test_mobile_dashboard_layout` | Verifies the page viewport size adheres to expected dimensions for mobile testing. |

## Part 3
**Objective**: Validate the full End-to-End lifecycle (API seed, UI validation, Mobile CSS, Tenant Isolation) for a single project flow.

**Scope**:
- Project creation via the API client.
- Web UI validation of project visibility for the creating tenant.
- Mobile web viewport emulation to verify responsive components (hamburger menu).
- Strict tenant isolation (verifying Company 1's project does not bleed into Company 2's DOM).
- Cleanup of test data.

**Out of Scope**:
- Live backend network resolution.
- Native mobile device testing.

**Test Environment**: Fully network-mocked. The `requests` library is mocked via `unittest.mock`, and Playwright UI requests to `/login` and `/dashboard` are intercepted and fulfilled with dynamic inline HTML via `page.route()`.

**Entry Criteria**: `company1_api_client` fixture loaded from `conftest.py`, and test data populated in `.env`.

**Exit Criteria / Pass Conditions**: Zero timeouts, mock API contract verification (`mock_post.call_args.kwargs`) strictly matches payload, and the tenant isolation assertion (`not_to_be_visible()`) passes unambiguously after route teardown.

**Test Cases**:
| Test Name | Description |
|-----------|-------------|
| `test_end_to_end_project_lifecycle` | Validates API project creation, UI rendering, mobile hamburger visibility, and tenant security isolation boundaries. |

## Known Limitations Across All Parts
- **Mobile app type**: Assumed to be mobile web. Native iOS/Android testing is unimplemented and would require Appium/BrowserStack integration.
- **BrowserStack**: Not currently implemented due to no account access. Emulation is local via Playwright `viewport` configs.
- **Test environment stability / Live Domain**: Tests run against a dummy domain (`app.workflowpro.com`). Part 1 skips live execution, and Part 3 fully mocks the network layer to remain deterministic without staging infrastructure.
- **Auth for automation**: Tokens (`TEST_API_TOKEN`) are injected via CI/environment variables rather than a dedicated token-minting backend endpoint.
- **Role permission matrix**: Fine-grained role permissions are assumed and mocked rather than verified against a fully documented live RBAC spec.
- **Test Cleanup Strategy**: The cleanup logic assumes `delete_project` API is idempotent and safe to call even if earlier steps failed.
